BAB 1

Pendahuluan & Latar Belakang Proyek

Nalarku.ai adalah platform belajar berbasis web yang dikembangkan secara mandiri, dirancang agar proses belajar mandiri menjadi lebih personal, interaktif, dan terukur. Berbeda dari platform belajar konvensional yang bersifat statis (materi tetap, soal tetap), Nalarku.ai dilengkapi dengan asisten kecerdasan buatan bernama Nala AI yang dapat berinteraksi langsung dengan pengguna, menjawab pertanyaan, membuatkan soal latihan, memberi umpan balik, serta mengakses berbagai fitur pendukung belajar seperti Pomodoro timer, sistem gamifikasi (XP dan leaderboard), dan plagiarism checker.

Karakteristik utama yang membedakan Nalarku.ai dari sekadar "aplikasi belajar dengan chatbot" adalah kemampuan Nala AI untuk mengakses tools — artinya AI tidak hanya menjawab dengan teks, tetapi dapat memicu fungsi-fungsi lain dalam sistem (memulai sesi Pomodoro, menghasilkan kuis baru, memeriksa originalitas tulisan pengguna) dan menyesuaikan responsnya berdasarkan riwayat dan progres belajar tiap pengguna secara personal.

1.1 Permasalahan yang Dijawab

Pembangunan kecerdasan buatan dari nol — termasuk melatih model bahasa besar (Large Language Model/LLM) sendiri — membutuhkan sumber daya komputasi, data, dan biaya yang sangat besar, jauh di luar jangkauan proyek mahasiswa maupun startup tahap awal. Oleh karena itu, Nalarku.ai mengambil pendekatan yang lazim digunakan industri: tidak membangun model AI sendiri, melainkan mengakses kecerdasan AI melalui Application Programming Interface (API) milik penyedia model pihak ketiga.

MENGAPA INI PENTING DIPRESENTASIKAN
Dosen penguji kemungkinan akan menanyakan: "Jadi AI-nya buatan kalian sendiri atau bukan?" Dokumen ini secara eksplisit menjelaskan bahwa kecerdasan inti (model bahasa) berasal dari penyedia API pihak ketiga, sementara nilai tambah dan orisinalitas proyek terletak pada arsitektur sistem, desain interaksi, personalisasi, integrasi tools, dan pengalaman belajar yang dibangun di atasnya — bukan pada pembuatan model AI itu sendiri. Kejujuran teknis ini justru memperkuat kredibilitas, karena membangun model sendiri di luar skema akademik bukanlah hal yang realistis maupun diharapkan pada level ini.

1.2 Tujuan Dokumen

Dokumen ini disusun untuk tiga tujuan:

  1. Menjelaskan secara teknis dan mendalam apa itu API, bagaimana cara kerjanya, dan secara khusus bagaimana API model AI bekerja — mulai dari konsep paling dasar hingga detail arsitektur.
  2. Mendokumentasikan arsitektur sistem Nalarku.ai secara menyeluruh, termasuk strategi penggunaan beberapa penyedia API (multi-provider/campuran) dan alasan di baliknya.
  3. Menyediakan skrip dan strategi presentasi yang dapat langsung digunakan tim dalam sidang atau presentasi proposal akademik, lengkap dengan antisipasi pertanyaan penguji.
BAB 2

Apa Itu API? Konsep Dasar dari Nol

2.1 Definisi dan Analogi Sehari-hari

API (Application Programming Interface) adalah seperangkat aturan dan mekanisme yang memungkinkan satu program perangkat lunak berkomunikasi dan meminta layanan dari program perangkat lunak lain, tanpa kedua program tersebut perlu mengetahui detail internal satu sama lain. API mendefinisikan apa yang bisa diminta, bagaimana cara memintanya, dan dalam format apa jawabannya akan diberikan.

ANALOGI: API SEBAGAI PRAMUSAJI RESTORAN
Bayangkan sebuah restoran. Dapur (sistem internal) memiliki resep dan proses memasak yang rumit, dan pelanggan (aplikasi yang meminta) tidak perlu — dan tidak boleh — masuk ke dapur untuk mengambil makanan sendiri. Sebagai gantinya, ada pramusaji (API) yang menerima pesanan dalam format tertentu (menu), meneruskannya ke dapur, lalu mengembalikan hasilnya ke meja pelanggan. Pelanggan tidak perlu tahu bagaimana cara memasak nasi goreng; ia cukup tahu cara memesannya lewat pramusaji. API bekerja dengan prinsip yang sama: ia menyembunyikan kerumitan internal sistem, dan hanya mengekspos "menu" — daftar permintaan yang diizinkan beserta formatnya.

Dalam konteks perangkat lunak modern, ketika dua sistem berbeda — katakanlah aplikasi web Nalarku.ai dan server penyedia model AI — perlu saling berkomunikasi melalui internet, mekanisme paling umum yang digunakan adalah Web API berbasis HTTP, sering disebut juga REST API (Representational State Transfer).

2.2 Cara Kerja API Berbasis HTTP/REST

HTTP (HyperText Transfer Protocol) adalah protokol komunikasi yang sama dengan yang digunakan browser untuk membuka halaman web. Saat sebuah aplikasi memanggil API, pada dasarnya ia mengirim pesan terstruktur ke sebuah alamat (URL/endpoint) di server, dan server membalas dengan pesan terstruktur lain.

KomponenPenjelasanContoh dalam Nalarku.ai
EndpointAlamat URL spesifik yang menjadi tujuan permintaanhttps://api.provider.com/v1/messages
MethodJenis aksi yang diminta (GET = ambil data, POST = kirim/buat data, dst.)POST — mengirim pertanyaan pengguna
HeaderMetadata permintaan, termasuk otentikasi (API key) dan format dataAuthorization: Bearer sk-xxxx
BodyIsi data yang dikirim, biasanya dalam format JSON{"message": "Jelaskan stratifikasi sosial"}
ResponseBalasan dari server, juga umumnya JSON{"answer": "Stratifikasi sosial adalah..."}
Status CodeKode tiga digit penanda hasil permintaan200 OK, 401 Unauthorized, 429 Too Many Requests

2.3 Anatomi Request dan Response

Berikut contoh sederhana bentuk permintaan (request) dan balasan (response) API secara umum, dalam format JSON yang menjadi standar de facto pertukaran data API modern:

// Contoh REQUEST yang dikirim aplikasi ke API
POST /v1/messages HTTP/1.1
Host: api.contohprovider.com
Content-Type: application/json
Authorization: Bearer <API_KEY>

{
  "model": "nama-model-tertentu",
  "messages": [
    { "role": "user", "content": "Apa itu fotosintesis?" }
  ],
  "max_tokens": 500
}
// Contoh RESPONSE yang dikembalikan API
HTTP/1.1 200 OK
Content-Type: application/json

{
  "id": "msg_01abcXYZ",
  "content": [
    { "type": "text", "text": "Fotosintesis adalah proses..." }
  ],
  "usage": { "input_tokens": 12, "output_tokens": 86 }
}
ISTILAH KUNCI: API KEY
API key adalah "kunci akses" unik berupa untaian karakter acak yang membuktikan identitas pemanggil API kepada penyedia layanan. Setiap permintaan ke API model AI wajib menyertakan API key di header, sehingga penyedia tahu siapa yang meminta, dapat menagih biaya penggunaan, dan dapat membatasi penyalahgunaan. API key harus diperlakukan seperti kata sandi — tidak boleh ditaruh di kode frontend yang bisa dilihat publik, melainkan hanya disimpan dan digunakan di sisi backend/server.
BAB 3

API dalam Konteks Kecerdasan Buatan (LLM API)

3.1 Apa yang Dimaksud "Model Bahasa Besar" (LLM)

Large Language Model (LLM) adalah model statistik berskala sangat besar yang dilatih menggunakan miliaran hingga triliunan kata teks, sehingga mampu memprediksi dan menghasilkan teks yang koheren, relevan, dan tampak "memahami" bahasa manusia. Model seperti inilah yang menjadi otak di balik kemampuan Nala AI menjawab pertanyaan, menjelaskan konsep, atau membuat soal kuis.

Yang membuat model ini sulit dan mahal untuk dibangun sendiri bukan algoritmanya yang rahasia, melainkan skala sumber daya yang dibutuhkan: ribuan unit pemroses grafis (GPU) kelas server yang berjalan selama berminggu-minggu hingga berbulan-bulan, dataset teks berskala internet yang telah dibersihkan dan dikurasi, serta tim peneliti khusus untuk proses fine-tuning dan keselamatan model. Inilah alasan mendasar mengapa hampir seluruh aplikasi AI di dunia — termasuk milik perusahaan besar sekalipun — memilih mengakses model lewat API, bukan melatih model sendiri.

3.2 Token, Context Window, dan System Prompt

Tiga istilah berikut adalah konsep yang akan sangat sering muncul saat menjelaskan API AI ke penguji, sehingga penting dipahami secara presisi:

a. Token

Model tidak membaca teks per huruf atau per kata utuh, melainkan memecahnya menjadi unit-unit kecil yang disebut token — bisa berupa satu kata pendek, sebagian kata, atau tanda baca. Sebagai gambaran kasar, satu token kurang lebih setara 4 karakter dalam bahasa Inggris, sedikit lebih banyak untuk Bahasa Indonesia. Penyedia API hampir selalu menagih biaya berdasarkan jumlah token yang digunakan, baik token masukan (input) maupun token keluaran (output).

b. Context Window

Context window adalah batas maksimum jumlah token (gabungan input dan output) yang dapat diproses model dalam satu kali permintaan. Ini penting bagi Nalarku.ai karena menentukan seberapa banyak riwayat percakapan atau materi belajar yang bisa "diingat" Nala AI dalam satu sesi — di luar batas tersebut, bagian percakapan terlama harus dipangkas atau diringkas oleh backend sebelum dikirim ulang ke API.

c. System Prompt

System prompt adalah instruksi tersembunyi yang disisipkan backend di awal setiap permintaan, tidak terlihat oleh pengguna, yang mengatur bagaimana model harus berperilaku — gaya bahasa, batasan topik, persona, hingga format jawaban. Inilah mekanisme utama yang membuat "Nala AI" terasa seperti karakter yang konsisten, padahal di baliknya adalah model generik yang sama yang diberi instruksi berbeda.

ANALOGI: SYSTEM PROMPT SEPERTI SKRIP UNTUK AKTOR
Bayangkan seorang aktor profesional yang bisa memerankan tokoh apa pun selama diberi skrip dan arahan sutradara. Model AI di balik Nala AI pada dasarnya adalah "aktor" generik yang sama dengan yang dipakai ribuan aplikasi lain. System prompt adalah skrip dan arahan sutradara yang membuatnya tampil sebagai "Nala AI" — ramah, fokus pada konteks belajar, menggunakan gamifikasi — alih-alih sekadar asisten generik tanpa karakter.

4.2 Strategi Multi-Provider (Pendekatan Campuran)

Nalarku.ai dirancang menggunakan lebih dari satu penyedia API AI secara bersamaan, bukan bergantung pada satu provider tunggal. Pendekatan ini, yang dalam industri sering disebut multi-provider atau model routing, dipilih dengan tiga pertimbangan utama:

1Optimasi tugas spesifik. Tidak semua model AI unggul di semua hal. Provider tertentu mungkin lebih baik untuk percakapan tutor yang panjang dan kontekstual, sementara provider lain lebih efisien dan murah untuk tugas terstruktur seperti menghasilkan soal kuis dalam format JSON, atau menghitung kemiripan teks untuk plagiarism checker.
2Efisiensi biaya. Model dengan kemampuan reasoning tinggi umumnya berbiaya lebih mahal per token. Dengan merutekan tugas-tugas sederhana (klasifikasi, validasi format) ke model yang lebih ringan dan murah, dan menyimpan model premium hanya untuk interaksi tutor yang membutuhkan kualitas tinggi, biaya operasional keseluruhan dapat ditekan signifikan.
3Ketahanan sistem (resilience). Jika satu penyedia API mengalami gangguan layanan (downtime) atau pembatasan kuota (rate limit), sistem dapat secara otomatis mengalihkan permintaan ke penyedia cadangan (mekanisme fallback), sehingga layanan Nalarku.ai tetap dapat berjalan tanpa titik kegagalan tunggal (single point of failure).

4.3 Lapisan Persona "Nala AI"

Penting dipahami dan disampaikan dengan jelas saat presentasi: "Nala AI" bukanlah model AI yang berdiri sendiri, melainkan sebuah lapisan logika (persona layer) di sisi backend yang membungkus model-model AI eksternal tersebut. Lapisan ini bertanggung jawab untuk:

  • Menyisipkan system prompt yang mendefinisikan kepribadian, gaya bicara, dan batasan topik Nala AI.
  • Menyuntikkan konteks personal pengguna — progres belajar, riwayat kuis, level XP — ke dalam setiap permintaan, sehingga jawaban terasa personal.
  • Menentukan kapan AI perlu "memanggil" tools tertentu, misalnya mendeteksi niat pengguna untuk memulai sesi belajar terjadwal lalu memicu fungsi Pomodoro.
  • Memilih provider API mana yang paling sesuai untuk jenis permintaan yang sedang diproses (lihat 4.2).
BAB 5

Apa yang Terjadi di Balik Layar Saat Pengguna Bertanya

Bab ini menjabarkan secara rinci, langkah demi langkah, apa yang sebenarnya terjadi secara teknis sejak pengguna mengetik pertanyaan kepada Nala AI hingga jawaban muncul di layar. Pemahaman alur ini krusial untuk dapat menjawab pertanyaan teknis mendalam dari penguji.

5.1 Diagram Alur Permintaan (Sequence Flow)

Gambar 5.1 — Alur permintaan dari pengguna hingga jawaban Nala AI ditampilkan kembali, melibatkan empat aktor: Pengguna, Frontend, Backend Nalarku, dan Provider API AI.

5.2 Penjelasan Tahap demi Tahap

1Input pengguna. Pengguna mengetik pertanyaan atau perintah di antarmuka chat dan menekan tombol kirim. Pada tahap ini, teks masih murni berupa string di sisi klien (browser).
2Permintaan ke backend. Frontend mengirimkan permintaan HTTP POST ke endpoint backend milik Nalarku.ai sendiri (bukan langsung ke API AI eksternal) — ini penting karena API key penyedia AI tidak boleh terekspos di sisi klien.
3Penyusunan prompt. Backend mengambil pesan pengguna, menggabungkannya dengan system prompt persona Nala AI, riwayat percakapan terbaru dari database, dan data konteks belajar pengguna (misalnya: "pengguna sedang mempelajari materi stratifikasi sosial, baru menyelesaikan 60% kuis bab ini"). Seluruh elemen ini disusun menjadi satu paket permintaan terstruktur.
4Pemanggilan API eksternal. Backend mengirim paket tersebut ke endpoint API milik penyedia model AI yang dipilih oleh AI Router, disertai header otentikasi berisi API key rahasia yang hanya disimpan di server.
5Pemrosesan oleh model. Di sisi server penyedia API, model AI memecah teks masukan menjadi token, memprosesnya melalui jaringan neural berlapis (mekanisme attention yang menilai relevansi antar-token), lalu menghasilkan jawaban secara bertahap, token demi token, berdasarkan probabilitas statistik kata berikutnya yang paling sesuai dengan konteks.
6Respons dikembalikan. Hasil jawaban dikemas dalam format JSON dan dikirim kembali melalui HTTP ke backend Nalarku.ai, disertai metadata seperti jumlah token yang digunakan (relevan untuk penghitungan biaya).
7Pascaproses di backend. Backend menyimpan riwayat percakapan ke database, memperbarui poin pengalaman (XP) atau status pencapaian (badge) bila relevan, dan memformat ulang jawaban sesuai skema yang dipahami antarmuka frontend.
8Tampil ke pengguna. Frontend menerima respons dan menampilkannya, umumnya dengan efek mengetik bertahap (streaming atau simulasi ketik) agar interaksi terasa lebih alami dan langsung (real-time).
KONSEP PENTING: API BERSIFAT STATELESS
Model AI tidak benar-benar "mengingat" percakapan sebelumnya antar permintaan terpisah. Setiap kali pengguna mengirim pesan baru, backend harus mengirim ULANG seluruh riwayat percakapan yang relevan sebagai bagian dari permintaan baru. Itulah sebabnya pengelolaan context window dan strategi peringkasan riwayat menjadi bagian penting dari desain backend Nalarku.ai — bukan keterbatasan, melainkan karakteristik mendasar dari cara kerja API model bahasa.
BAB 6

Implementasi Teknis & Contoh Kode

6.1 Contoh Request API (Sisi Backend)

Berikut contoh implementasi pemanggilan API model AI dari backend Node.js, yang merepresentasikan Langkah 4 pada Bab 5. Catatan: nama provider, model, dan endpoint dalam contoh ini bersifat generik/ilustratif sehingga dapat disesuaikan dengan provider final yang dipilih tim.

// backend/services/aiRouter.js
async function kirimKeProviderAI(payload, providerConfig) {
  const response = await fetch(providerConfig.endpoint, {
    method: "POST",
    headers: {
      "Content-Type": "application/json",
      "Authorization": `Bearer ${process.env[providerConfig.apiKeyEnv]}`
    },
    body: JSON.stringify({
      model: providerConfig.model,
      messages: payload.messages,
      max_tokens: 800,
      temperature: 0.7
    })
  });

  if (!response.ok) {
    throw new Error(`Provider error: ${response.status}`);
  }

  return await response.json();
}

6.2 Desain System Prompt untuk Nala AI

System prompt adalah bagian paling menentukan karakter Nala AI. Berikut struktur konseptual yang dapat dijelaskan ke penguji sebagai bagian dari "rancangan kepribadian AI" tanpa perlu membocorkan teks lengkap sistem produksi:

// backend/prompts/nalaPersona.js
function susunSystemPrompt(konteksUser) {
  return `
Kamu adalah Nala AI, asisten belajar dalam platform Nalarku.ai.
Gaya bicara: ramah, suportif, memotivasi, menggunakan bahasa
yang mudah dipahami pelajar Indonesia.

Konteks pengguna saat ini:
- Nama: ${konteksUser.nama}
- Materi yang sedang dipelajari: ${konteksUser.materiAktif}
- Progres kuis: ${konteksUser.progresKuis}%
- Level XP: ${konteksUser.levelXP}

Batasan:
- Jangan menjawab di luar topik akademik/edukasi.
- Jika pengguna tampak frustrasi, berikan dorongan motivasi
  sebelum melanjutkan materi.
- Sarankan sesi Pomodoro jika pengguna sudah belajar lebih
  dari 45 menit tanpa jeda.
`;
}
CATATAN AKADEMIK
Bagian inilah yang layak ditekankan sebagai kontribusi orisinal proyek dalam laporan/skripsi: bukan algoritma di dalam model AI (yang merupakan karya penyedia API), melainkan rancangan interaksi, desain prompt, manajemen konteks personal, dan logika integrasi tools yang membungkus model generik menjadi pengalaman belajar "Nala AI" yang khas dan personal.

6.3 Penanganan Kegagalan (Fallback)

Sebagai konsekuensi dari strategi multi-provider pada Bab 4, sistem perlu menangani skenario ketika satu provider gagal merespons:

// backend/services/aiRouter.js
async function prosesPertanyaan(payload) {
  const urutanProvider = ["providerUtama", "providerCadangan"];

  for (const nama of urutanProvider) {
    try {
      return await kirimKeProviderAI(payload, konfigProvider[nama]);
    } catch (err) {
      console.warn(`${nama} gagal, mencoba provider berikutnya...`);
      continue;
    }
  }

  throw new Error("Semua provider AI tidak tersedia saat ini.");
}
BAB 7

Pertimbangan Keamanan, Biaya, dan Skalabilitas

7.1 Keamanan API Key

  • Tidak pernah disimpan di frontend. API key hanya hidup di environment variable server backend, tidak pernah dikirim ke browser dalam bentuk apa pun.
  • Tidak pernah di-commit ke repository. Disimpan dalam file .env yang dimasukkan ke .gitignore, dengan nilai aktual hanya dikonfigurasi langsung di platform hosting (mis. Vercel Environment Variables).
  • Proxy pattern. Seluruh permintaan ke API AI selalu melalui backend Nalarku.ai sebagai perantara — frontend tidak pernah memanggil API provider secara langsung.

7.2 Estimasi & Kontrol Biaya

Karena penyedia API umumnya menagih berdasarkan jumlah token, beberapa strategi kontrol biaya yang relevan untuk diterapkan dan disebutkan dalam presentasi:

StrategiTujuan
Membatasi max_tokens per responsMencegah jawaban yang tidak perlu panjang menghabiskan kuota
Meringkas riwayat percakapan lamaMengurangi token input yang dikirim ulang setiap permintaan
Merutekan tugas sederhana ke model lebih murahHanya memakai model premium saat dibutuhkan kualitas tinggi
Caching jawaban untuk pertanyaan umumMenghindari pemanggilan API berulang untuk pertanyaan identik
Rate limiting per penggunaMencegah penyalahgunaan yang membengkakkan biaya operasional

7.3 Skalabilitas

Karena beban komputasi berat (menjalankan model AI) sepenuhnya berada di sisi penyedia API, bukan di server Nalarku.ai sendiri, beban utama yang perlu diskalakan oleh tim hanyalah lapisan backend (routing, autentikasi, database) — yang jauh lebih ringan dibanding mengoperasikan infrastruktur model AI. Ini merupakan keuntungan arsitektural tambahan dari pendekatan berbasis API yang dapat disampaikan sebagai justifikasi desain sistem.

BAB 8

Skrip Presentasi Sidang

Bagian ini berisi naskah presentasi yang dapat dibawakan langsung, disusun agar mengalir secara natural namun tetap menunjukkan penguasaan teknis. Skrip dibagi per segmen dengan estimasi durasi, dan dirancang untuk dibawakan dengan bahasa sendiri — bukan dihafalkan kata per kata.

8.1 Segmen Pembukaan

PEMBICARA · ~1-2 MENIT

"Selamat pagi/siang, Bapak/Ibu dosen penguji. Perkenalkan, kami akan mempresentasikan Nalarku.ai — sebuah platform belajar berbasis web yang dilengkapi asisten kecerdasan buatan personal bernama Nala AI.

Yang membedakan Nalarku.ai dari platform belajar pada umumnya adalah: Nala AI tidak sekadar menjawab pertanyaan secara generik, tetapi dapat memahami konteks belajar setiap pengguna secara personal, serta dapat mengakses fitur-fitur lain dalam sistem — seperti memicu sesi Pomodoro, menghasilkan kuis adaptif, dan memeriksa orisinalitas tulisan pengguna.

Dalam presentasi ini, kami akan menjelaskan bagaimana kecerdasan buatan ini bekerja secara teknis, arsitektur sistem yang kami bangun, dan keputusan-keputusan desain di baliknya."

8.2 Segmen Penjelasan Konsep API

PEMBICARA · ~3-4 MENIT

"Sebelum masuk ke arsitektur, kami ingin menjelaskan satu fondasi penting: bagaimana Nalarku.ai mendapatkan kecerdasan AI-nya.

Membangun model bahasa besar dari nol — melatih AI seperti yang digunakan ChatGPT atau Claude — membutuhkan ribuan unit GPU server, dataset berskala internet, dan biaya riset yang jauh di luar skala proyek mahasiswa, bahkan startup tahap awal sekalipun. Karena itu, pendekatan yang kami ambil — dan ini adalah pendekatan standar industri, digunakan oleh hampir semua aplikasi berbasis AI saat ini — adalah mengakses kecerdasan tersebut melalui Application Programming Interface, atau API.

Secara sederhana, API adalah perantara yang memungkinkan sistem kami 'bertanya' kepada model AI milik penyedia pihak ketiga melalui internet, lalu menerima jawabannya kembali, tanpa kami perlu tahu — atau perlu membangun — cara kerja model itu sendiri di baliknya. Kami mengirim permintaan terstruktur berisi pertanyaan pengguna, dan menerima balasan terstruktur berisi jawaban AI, dalam hitungan detik.

Penting kami tegaskan: kecerdasan inti — model bahasanya sendiri — bukan buatan kami. Yang menjadi kontribusi dan orisinalitas kami adalah bagaimana kami merancang sistem di sekitarnya: bagaimana AI tersebut diberi 'kepribadian' sebagai Nala AI, bagaimana ia dipersonalisasi untuk tiap pengguna, dan bagaimana ia diintegrasikan dengan fitur-fitur belajar lainnya."

8.3 Segmen Penjelasan Arsitektur Sistem

PEMBICARA · ~4-5 MENIT, SAMBIL MENAMPILKAN DIAGRAM BAB 4 & 5

"Berikut adalah arsitektur sistem kami secara keseluruhan. [Tunjuk diagram] Saat pengguna mengetik pertanyaan di antarmuka Nalarku.ai, permintaan tersebut tidak langsung dikirim ke penyedia AI, melainkan terlebih dahulu masuk ke backend kami sendiri.

Backend inilah yang menjadi 'otak orkestrasi' sistem kami. Ia melakukan tiga hal penting: pertama, menyusun konteks — menggabungkan pertanyaan pengguna dengan riwayat percakapan, progres belajar, dan instruksi kepribadian Nala AI. Kedua, memilih penyedia API mana yang paling sesuai untuk jenis permintaan tersebut — kami menyebutnya AI Router — karena kami menggunakan lebih dari satu penyedia API secara bersamaan, bukan hanya satu.

Mengapa lebih dari satu? Karena setiap penyedia memiliki kekuatan, kecepatan, dan struktur biaya berbeda. Untuk percakapan tutor yang membutuhkan pemahaman kontekstual mendalam, kami merutekan ke model yang lebih kuat. Untuk tugas terstruktur seperti menghasilkan soal kuis, kami dapat menggunakan model yang lebih ringan dan efisien biaya. Pendekatan campuran ini juga memberi ketahanan sistem — jika satu penyedia mengalami gangguan, sistem secara otomatis beralih ke penyedia cadangan.

Setelah jawaban diterima dari API, backend melakukan pascaproses: menyimpan riwayat, memperbarui poin pengalaman pengguna, lalu mengirimkan hasil akhir ke antarmuka pengguna."

8.4 Segmen Penjelasan "Apa yang Terjadi di Balik Layar"

PEMBICARA · ~2-3 MENIT, OPSIONAL JIKA WAKTU TERBATAS

"Untuk memberi gambaran lebih konkret, izinkan kami menjelaskan apa yang terjadi dalam hitungan detik setelah pengguna menekan tombol kirim. [Tunjuk diagram sequence flow] Pesan pengguna dikirim ke backend kami, backend menyusun paket permintaan lengkap dengan konteks personal pengguna, lalu mengirimkannya ke API model AI disertai kunci otentikasi yang tersimpan aman di server — tidak pernah terekspos ke pengguna.

Di sisi penyedia API, model memproses teks tersebut menjadi unit-unit kecil yang disebut token, lalu menghasilkan jawaban secara bertahap berdasarkan pemahaman statistiknya terhadap bahasa. Jawaban ini kemudian kembali ke backend kami, disimpan ke riwayat, dan ditampilkan ke pengguna — seluruh proses ini umumnya berlangsung dalam satu hingga lima detik."

8.5 Antisipasi Pertanyaan Dosen Penguji

Bagian ini krusial — penguji kemungkinan besar akan menguji pemahaman mendalam, bukan sekadar hafalan presentasi. Berikut pertanyaan yang paling mungkin muncul beserta kerangka jawaban:

Q: "Jadi AI-nya bukan buatan kalian sendiri? Lalu apa kontribusi ilmiah/orisinal proyek ini?"
A: Tegaskan bahwa hampir seluruh aplikasi berbasis AI di industri — termasuk produk-produk besar — menggunakan pendekatan serupa: memanfaatkan API model fondasi, lalu membangun nilai tambah di atasnya. Kontribusi orisinal proyek terletak pada: (1) arsitektur sistem multi-provider dan strategi routing, (2) desain persona dan system prompt Nala AI, (3) sistem personalisasi berbasis konteks belajar pengguna, (4) integrasi AI dengan tools pembelajaran (gamifikasi, Pomodoro, plagiarism checker), dan (5) pengalaman pengguna end-to-end yang dirancang khusus untuk konteks pendidikan Indonesia.
Q: "Bagaimana jika API yang kalian gunakan tiba-tiba menaikkan harga atau berhenti beroperasi?"
A: Jelaskan strategi multi-provider sebagai mitigasi risiko ini — arsitektur backend dirancang agar provider dapat ditukar atau ditambah tanpa mengubah logika aplikasi inti, karena lapisan persona dan logika bisnis dipisahkan dari pemanggilan API spesifik (prinsip abstraksi/decoupling).
Q: "Bagaimana keamanan data pengguna, mengingat data dikirim ke pihak ketiga?"
A: Jelaskan bahwa hanya data yang relevan dengan konteks pertanyaan yang dikirim (bukan seluruh database), API key dan kredensial tidak pernah terekspos, komunikasi menggunakan HTTPS terenkripsi, dan penyedia API yang dipilih tunduk pada kebijakan privasi serta perjanjian pemrosesan data standar industri. Jika relevan, sebutkan bahwa kebijakan retensi/penggunaan data dari tiap provider perlu ditinjau dan dicantumkan dalam kebijakan privasi aplikasi.
Q: "Apa bedanya Nala AI dengan ChatGPT biasa yang dibuka di browser?"
A: Tekankan personalisasi dan integrasi: Nala AI mengetahui progres belajar spesifik pengguna, riwayat kuis, dan dapat memicu aksi nyata dalam aplikasi (memulai Pomodoro, mencatat XP), sesuatu yang tidak dimiliki chatbot generik berdiri sendiri yang diakses terpisah dari platform belajar.
Q: "Berapa biaya operasional untuk menjalankan fitur AI ini?"
A: Jelaskan model biaya berbasis token (lihat Bab 7), dan strategi kontrol biaya yang diterapkan: pembatasan panjang respons, peringkasan riwayat, routing tugas sederhana ke model lebih murah, dan rate limiting. Jika sudah punya angka estimasi nyata dari testing, sampaikan sebagai data pendukung.
Q: "Apakah AI ini bisa memberikan jawaban yang salah? Bagaimana mengatasinya?"
A: Akui secara jujur bahwa model bahasa dapat menghasilkan informasi yang tidak akurat (dikenal sebagai fenomena hallucination dalam literatur AI), dan jelaskan langkah mitigasi yang relevan: system prompt yang membatasi topik pada materi akademik, kemungkinan validasi jawaban terhadap sumber materi resmi platform, serta umpan balik pengguna untuk perbaikan berkelanjutan.

BAB 9

Kesimpulan

Nalarku.ai dibangun di atas prinsip arsitektur yang jelas: memanfaatkan Application Programming Interface (API) sebagai jembatan untuk mengakses kecerdasan model bahasa besar dari penyedia pihak ketiga, alih-alih membangun dan melatih model AI sendiri — sebuah pendekatan yang realistis secara sumber daya dan selaras dengan praktik umum industri perangkat lunak berbasis AI.

Nilai tambah dan orisinalitas proyek terletak pada lapisan yang dibangun di atas fondasi tersebut: arsitektur backend dengan strategi multi-provider untuk efisiensi biaya dan ketahanan sistem, desain persona "Nala AI" melalui rekayasa system prompt, mekanisme personalisasi berbasis konteks belajar pengguna, serta integrasi AI dengan ekosistem fitur pembelajaran seperti gamifikasi, Pomodoro timer, dan plagiarism checker.

Pemahaman menyeluruh terhadap cara kerja API — mulai dari konsep dasar request-response, anatomi token dan context window, hingga implikasi keamanan dan biaya — menjadi bekal penting bagi tim untuk mempertanggungjawabkan keputusan desain sistem secara teknis maupun akademik di hadapan dosen penguji.
